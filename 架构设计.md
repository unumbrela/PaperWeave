# PaperWeave 架构设计

一个本地优先的论文精读与研究推进工作台，把一篇论文从「找到」到「读懂、想清、写下、画出」的全过程，收拢进同一条工作流。

```
检索 → 精读 → 梳理 → 立论 → 撰写 → 制图
```

---

## 一、技术选型

| 层 | 选型 |
| --- | --- |
| 前端 | Next.js 16（App Router）· React 19 · TypeScript · Tailwind CSS v4 |
| 后端 | Node.js（Next.js Route Handlers，约 28 个 API 路由） |
| 数据库 | PostgreSQL（Supabase 托管，含 Auth + 行级安全 RLS）；浏览器端 IndexedDB（Dexie）作本地缓存层 |
| 外部服务 | LLM（DeepSeek / OpenAI / Gemini 三级 fallback）· 文献源（OpenAlex / arXiv / Crossref / Europe PMC） |
| 可视化 | D3.js · Three.js · TensorFlow.js（浏览器内推理） |

---

## 二、总体架构（分层）

```
┌─────────────────────────────────────────────────────────┐
│  前端  React 页面 / 组件 / Hooks                          │
│  ──────────────────────────────────────────────────────  │
│   · UI 一律经「仓储层」读写论文数据，不直接 fetch 数据接口  │
└───────────────┬───────────────────────┬─────────────────┘
                │                        │
        仓储层 repository.ts        Route Handlers（后端）
                │                        │ 检索聚合 / AI 编排 /
        ┌───────┴───────┐                │ PDF 代理 / 分享 / 指标
        │               │                │
   IndexedDB        云同步层          外部服务
   (本地真相源)   cloud-sync.ts       LLM · 文献源
        │          (登录后镜像)            
        │               │                
        └──────────►  PostgreSQL（Supabase + RLS）
                     跨设备同步 · 按 user_id 隔离
```

**三层职责**

- **前端**：页面与交互组件。所有论文 / 标注 / 笔记 / 进度的读写都只调用**仓储层**（`lib/db/repository.ts`），不在 UI 里直连数据接口——从根上避免「本地有云端没有」的脏状态。
- **后端**（Node.js Route Handlers）：做编排，不持有业务状态。职责为多源文献检索聚合、LLM 调用与流式输出、PDF 代理（带 SSRF 防护）、只读分享、指标。公开免 key 路由统一限流。
- **数据库**：PostgreSQL 为跨设备**云端真相源**（登录后生效，RLS 保证每人只见自己的行）；IndexedDB 为**本地缓存层**，保证零配置、可离线。

---

## 三、数据流（核心心智模型）

写操作**先落本地、再异步镜像云端**，云端失败不影响本地：

```
UI/Hook ──► repository.ts ──► IndexedDB（即时落库，可离线）
                  └────────► cloudSync.push*()  // 登录后 fire-and-forget
                                    └──► PostgreSQL（带 user_id，RLS 隔离）
```

- **未登录 / 未配置**：`cloudSync` 全部 no-op，退化为纯本地模式，应用完整可用。
- **登录后**：每次写镜像到云端；登录瞬间先上推本地、再拉云端合并，换设备 / 清缓存即恢复。冲突按 `updated_at` 最后写入者优先。

---

## 四、数据库设计

5 张用户私有表（均带 `user_id` + RLS），以 `papers` 为中心：

```
papers ──┬──< annotations    （PDF 批注）
（论文）  ├──< research_notes  （研究笔记）
         ├──< sticky_notes    （页面便签）
         └──< read_progress   （阅读进度）
```

另有 2 张服务端共享表（仅 service-role 可读写、不对用户开放）：`search_cache`（检索结果缓存）、`shares`（只读分享快照）。完整 DDL 见 [`supabase/schema.sql`](supabase/schema.sql)。

> 注：PDF 文件二进制只存本地 IndexedDB，不入云端（仅同步元数据）。

## 五、后端 API 分类

| 类别 | 代表路由 | 说明 |
| --- | --- | --- |
| 文献检索 | `/api/paper-search` | 4 源并发聚合、去重、缓存 |
| AI 工作流 | `/api/idea-generator`、`/api/paper-writer`、`/api/compare-papers` | LLM 流式 / 非流式编排 |
| 文库问答 | `/api/library-qa/*` | RAG：embedding 索引 + 检索归纳 |
| 资源代理 | `/api/pdf-proxy` | PDF 抓取，域名白名单防 SSRF |
| 分享 / 指标 | `/api/share/*`、`/api/metrics` | 只读快照、访问指标聚合 |

LLM 接入分流式 / 非流式两路，均按 `DeepSeek → OpenAI → Gemini` 顺序自动 fallback；支持访客自带 key（BYOK，仅存其本机，服务端不持久化）。

---

## 六、关键设计原则

1. **单一真相源**：UI 不直连数据接口，论文数据读写全部经仓储层。
2. **本地优先 + 可选云同步**：零配置可离线运行；登录后透明升级为跨设备同步。
3. **零配置构建**：不配任何环境变量也能完整构建与运行，外部依赖一律惰性 / 可空降级。
4. **best-effort 外围**：缓存、云同步、指标失败一律吞掉，绝不影响主功能。
5. **纯函数优先**：构图、引文、RAG 检索、限流等可测逻辑抽为无副作用纯函数，路由只做编排。
